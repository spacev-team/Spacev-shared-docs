# AI와 더 친하게 지내는법 발표 스크립트

부제: 해야 할 것과 하지 말아야 할 것을 설정하여 내 기준대로 일하는 AI 만들기

발표 스타일: 문장을 외우기보다, 각 페이지의 핵심 키워드만 보고 자연스럽게 이어가기.

전체 흐름:
LLM과 모델 구분 → 좋은 프롬프트 → 기준을 저장하는 개념들 → 자주 생기는 문제 → 도구 연결 → 내 업무에 적용

- 목표는 AI를 답변 창이 아니라, 내 기준을 이해하는 작업 파트너로 만드는 것.

다음 연결:

## 04. LLM, 서비스, 모델 구분하기

키워드: LLM / 서비스는 입구 / 모델은 엔진 / 생각의 정도 / 비용

말할 흐름:
- LLM은 대화 맥락을 읽고 다음 답을 만들어내는 언어 모델.
- ChatGPT와 Claude는 서비스 이름이고, 실제 결과 품질은 안에서 선택한 모델에 영향을 받음.
- 그래서 “어떤 서비스가 좋아요?”보다 “어떤 모델이 이 일에 맞아요?”가 더 정확한 질문.
- 생각의 정도를 높이면 더 꼼꼼해질 수 있지만, 시간과 비용이 늘어남.

다음 연결:
- 좋은 모델을 골라도 요청이 흐리면 결과가 흐림. 그래서 좋은 프롬프트 기준으로 넘어감.

## 05. 좋은 프롬프트란?

키워드: 역할 / 기준 / 확인 / 같은 요청 비교 / 매번 쓰기 어려움

말할 흐름:
- 좋은 프롬프트는 길게 쓰는 것이 아니라 판단 기준이 있는 프롬프트.
- 같은 요청이라도 역할, 기준, 확인이 있으면 AI가 무엇을 보고 판단할지 분명해짐.
- 나쁜 예시는 “써줘”만 있고, 좋은 예시는 어떤 관점인지, 무엇을 포함하고 피할지, 마지막에 무엇을 확인할지 있음.
- 하지만 매번 이렇게 모든 기준을 쓰는 것은 비효율적.

다음 연결:
- 그래서 기준을 매번 입력하지 않도록 저장하는 개념들이 필요해짐.

## 06. AI를 이루는 기본 요소

키워드: Part 1 / 기준 저장 / 기억 / 절차 / 역할 / 맥락

말할 흐름:
- 이제 좋은 프롬프트 안에 들어가던 기준들이 실제 AI 도구에서는 어떤 형태로 나뉘는지 봄.
- 지침, 기억, 절차, 역할, 맥락을 구분하면 AI를 훨씬 안정적으로 다룰 수 있음.

다음 연결:

## 07. 먼저 볼 개념들

키워드: Instruction / Memory / Skill / Agent / Context / Context Window

말할 흐름:
- 지금은 세부 설명보다 이름과 역할만 잡는 페이지.
- Instruction은 항상 지켜야 하는 기준.
- Memory는 반복해서 설명하기 어려운 취향과 기준.
- Skill은 반복 절차, Agent는 역할, Context는 대화 흐름, Context Window는 참고 범위.

다음 연결:
- 첫 번째로, AI가 항상 지켜야 할 기준인 Instruction부터 봄.

## 08. Instruction(지침)

키워드: 항상 지켜야 하는 기준 / 역할 / 해야 할 것 / 하지 말아야 할 것 / 총 주방장

말할 흐름:
- Instruction은 AI가 사용자의 질문을 분석하기 전에 가장 먼저 읽는 운영 원칙.
- 총 주방장 예시로 보면, 전체 기준을 정하고 담당 Agent에게 일을 나누는 역할.
- 해야 할 일은 주문 조건 확인, 담당 나누기, 최종 점검.
- 하지 말아야 할 일은 금지 재료 무시, 임의 추가, 검토 없이 전달.

다음 연결:
- 지침이 반드시 지킬 운영 기준이라면, 다음은 반복해서 참고하면 좋은 기억.

## 09. Memory(기억)

키워드: 반복 기억 / 새 대화 / 취향 / 반복 기준 / 필수 조건은 재확인

말할 흐름:
- Memory는 새 대화를 열 때마다 똑같이 설명하기 번거로운 기준과 취향을 저장하는 곳.
- 예를 들면 쉬운 말 선호, 익숙한 예시 선호, 결과 정리 순서 같은 것.
- 단, 알레르기나 금지 재료처럼 중요한 조건은 Memory에만 맡기면 위험.
- Instruction은 먼저 읽는 운영 기준이고, Memory는 반복 설명을 줄이는 참고 기억이라는 차이를 짚음.
- 중요한 조건은 요청에서도 다시 확인하는 습관이 필요.

다음 연결:
- 이제 기억이 아니라 “반복되는 작업 순서”를 저장하는 Skill로 이동.

## 10. Skill(업무 절차)

키워드: 반복 실행 절차 / 시작 조건 / 진행 순서 / 확인 기준 / 결과 형식

말할 흐름:
- Skill은 반복되는 일을 같은 순서와 기준으로 실행하기 위한 절차.
- 시작 조건, 진행 순서, 확인 기준, 결과 형식을 정해두는 것이 핵심.
- 자료 정리, 회의록 요약, 안내문 작성처럼 매번 비슷한 순서로 하는 일에 유용함.
- 짧게 요청해도 정해둔 순서와 결과 형식이 반복되는 것이 장점.

다음 연결:
- Skill이 실행 순서라면, Agent는 그 일을 맡아 판단하고 책임지는 담당자.

## 11. Agent(역할 담당자)

키워드: 역할 담당자 / 판단 기준 / 위임 범위 / 결과 책임 / 담당 Agent

말할 흐름:
- Agent는 특정 역할과 책임을 맡은 AI 담당자.
- 한 사람이 다 하기보다 자료 조사, 초안 작성, 검토처럼 역할을 나누면 더 좋은 일에 사용.
- 핵심은 맡을 역할, 판단 기준, 위임 범위, 결과 책임이 정해져 있다는 점.
- Skill은 어떤 순서로 할지, Agent는 누가 어떤 관점으로 판단하고 책임질지를 정함.

다음 연결:
- 이제 AI가 지금 대화에서 무엇을 참고하는지, Context로 넘어감.

## 12. Context(대화의 흐름)

키워드: 질문과 답변 / 수정 요청 / 첨부 자료 / 대화 맥락 / 이슈의 원인

말할 흐름:
- Context는 지금까지 AI와 주고받은 말의 흐름.
- 질문, 답변, 다시 고쳐달라는 말, 첨부한 자료까지 포함.
- AI 사용 중 생기는 많은 문제는 필요한 맥락이 빠졌거나 오래된 맥락을 붙잡을 때 생김.
- 그래서 중요한 조건은 새 요청 첫 줄에 다시 적는 것이 좋음. 예: “조건 유지: 대상은 고객, 3문장 이하, 다음 행동 포함.”

다음 연결:
- 그런데 AI가 모든 맥락을 무제한으로 들고 가는 것은 아님. 참고 범위가 있음.

## 13. Context Window(참고 범위)

키워드: 참고 막대 / 대화 누적 / 요약 / 데이터 유실 / 중요한 기준 재입력

말할 흐름:
- Context Window는 AI가 한 번에 참고할 수 있는 정보의 범위.
- 대화가 길어지면 이전 내용이 그대로 남는 것이 아니라 요약될 수 있음.
- 요약 과정에서 세부 조건, 표현, 작은 기준은 빠질 수 있음.
- 중요한 기준은 새 요청 앞에 다시 적으면 현재 참고 범위에서 더 강하게 반영됨.

다음 연결:
- 여기까지 봐도 처음부터 완벽한 결과가 나오지는 않음. 반복 개선이 필요.

## 14. 처음부터 완벽한 결과물이 만들어지지는 않습니다

키워드: 첫 결과 / 피드백 / 저장 / 다음 작업 / Codex에게 맡기기

말할 흐름:
- AI는 처음부터 내 기준을 완벽하게 알지 못함.
- 결과가 너무 어렵거나, 근거가 약하거나, 예시가 안 맞으면 피드백을 줌.
- 그 피드백을 Instruction, Memory, Skill로 옮기면 다음 결과가 좋아짐.
- 직접 다 외워서 설정하려고 하지 말고, Codex에게 설정 파일로 바꿔달라고 시킬 수 있음.

다음 연결:
- 이제 실제로 우리가 자주 겪는 불편한 사례를 개념과 연결해 봄.

## 15. 불편했던 사례로 다시 보기

키워드: Part 2 / 말 안 들음 / 까먹음 / 자신 있게 틀림 / 원인 진단

말할 흐름:
- 이제부터는 개념 설명이 아니라 실제로 많이 겪는 장면.
- “왜 자꾸 말을 안 듣지?”, “왜 까먹지?”, “왜 틀렸는데 자신 있지?”를 하나씩 봄.
- 목적은 AI를 탓하기보다 어디에 기준을 넣어야 하는지 찾는 것.

다음 연결:
- 첫 번째 사례는 하지 말라고 했는데 다시 하는 상황.

## 16. “하지 말라고 했는데 자꾸 말을 안 들어요.”

키워드: 금지 표현 / Instruction / Prompt / 검수 요청 / 반복 금지선

말할 흐름:
- “변명처럼 쓰지 말라”고 했는데 다시 비슷한 표현이 나오는 상황.
- 한 번의 Prompt에만 적은 조건은 대화가 이어지며 약해질 수 있음.
- 계속 피해야 하는 표현은 Instruction에 저장.
- 이번 답변에서 중요한 조건은 Prompt 앞에도 다시 적고, 마지막에는 검수 요청.

다음 연결:
- 두 번째는 말했는데도 기억하지 못하는 것처럼 보이는 상황.

## 17. “분명히 말했는데 또 까먹었어요.”

키워드: 3문장 이하 / 기억 무제한 아님 / Memory / Context Window / 새 요청 앞 기준

말할 흐름:
- AI의 기억은 사람처럼 모든 것을 무제한으로 보존하는 방식이 아님.
- 대화가 길어지면 조건이 섞이거나 요약되며 약해질 수 있음.
- 반복되는 취향은 Memory에 저장.
- 이번 답변의 필수 조건은 “3문장 이하, 정중한 말투”처럼 새 요청 앞에 다시 적음.

다음 연결:
- 세 번째는 틀렸는데도 너무 자연스럽게 말하는 상황.

## 18. “틀렸는데 너무 자신 있게 말해요.”

키워드: 그럴듯한 답 / 근거 없음 / 추측 표시 / 최신 정보 / 확인 Skill

말할 흐름:
- AI는 정답 창고에서 꺼내기보다, 문맥상 자연스러운 답을 만들어내는 방식.
- 근거 자료가 없으면 그럴듯한 추측이 섞일 수 있음.
- “확인 못 한 내용은 추측이라고 표시”를 Instruction에 넣음.
- 반복적으로 중요한 일은 근거 확인 → 추측 분리 → 체크리스트 순서의 Skill로 만들 수 있음.

다음 연결:
- 이제 답변을 잘 받는 수준을 넘어, 외부 도구와 연결되는 단계로 넘어감.

키워드: Part 3 / 답변 창을 넘어 / 외부 도구 / 실제 결과물 / 제작형 AI

- 이제는 AI가 외부 도구와 연결되고, 실제 파일과 결과물을 만드는 흐름.
- MCP, Plugin, Codex, Claude Code는 이 단계에서 중요한 개념.

다음 연결:

## 20. MCP란?

키워드: 연결 표준 / AI → 명령어 → MCP → 산출물 / Notion / Figma / Slack

말할 흐름:
- MCP는 AI가 외부 서비스의 데이터와 기능을 정해진 방식으로 다루게 하는 연결 표준.
- 사용자의 명령어가 MCP를 거쳐 실제 산출물로 바뀜.
- Notion은 업무 목록 정리, Figma는 화면 문구 점검, Slack은 이슈 공유처럼 연결 가능.
- 핵심은 AI가 혼자 답하는 것을 넘어 외부 도구를 다룰 수 있다는 점.

다음 연결:
- MCP 같은 연결, Skill, Agent, Tool을 묶어서 가져다 쓰는 것이 Plugin.

## 21. PlugIn 이란?

키워드: 기능 묶음 / 꽂아 쓰기 / MCP / Skill / Agent / Tool

말할 흐름:
- Plugin은 AI에 기능을 꽂아 쓰는 묶음.
- AI 자체를 바꾸는 것이 아니라 필요한 도구, 절차, 역할을 연결하는 방식.
- 하나의 Plugin 안에 MCP, Skill, Agent, Tool이 함께 들어갈 수 있음.
- 그래서 답변만 받는 것이 아니라 실제 작업까지 맡길 수 있음.

다음 연결:
- 이제 ChatGPT/Claude와 Codex/Claude Code의 차이를 짚고, 왜 작업형 AI를 쓰는지 봄.

## 22. ChatGPT/Claude와 Codex/Claude Code의 차이

키워드: 대화형 AI / 작업형 AI / 채팅창 / 내 컴퓨터 / 대신 실행 / 결과물

말할 흐름:
- ChatGPT와 Claude는 채팅창에서 답변, 정리, 초안을 도와주는 대화형 AI에 가까움.
- Codex와 Claude Code는 내 컴퓨터의 파일, 프로젝트, 명령어를 다루며 실제 작업을 진행하는 작업형 AI에 가까움.
- 그래서 사용하는 이유는 AI를 챗봇으로만 쓰는 것을 넘어, 나 대신 컴퓨터에서 반복 작업을 실행하게 하기 위해서.
- 핵심은 대화에서 끝나는 것이 아니라 실행과 결과물로 이어진다는 점.

다음 연결:
- 그럼 실제로 어떤 결과물까지 만들 수 있는지 사례를 봄.

## 23. Codex와 Claude Code로 어디까지 할 수 있을까?

키워드: 교육자료 / 발표자료 / 업무 파일 / 스프레드시트 / 슬라이드

말할 흐름:
- 지금 보고 있는 이 교육자료도 AI와 함께 구조, 문장, UI, 발표 흐름을 계속 다듬은 결과물.
- 업무 파일도 만들 수 있음. 예를 들면 스프레드시트, 구글 슬라이드, 문서 같은 결과물.
- 중요한 건 “AI가 내 컴퓨터를 조작한다”가 아니라, 최종적으로 쓸 수 있는 결과물을 만든다는 점.

다음 연결:
- 문서와 파일을 넘어, 내 업무에 맞는 작은 도구도 만들 수 있음.

## 24. 내 업무 도구와 Figma PlugIn도 만들 수 있습니다

키워드: 개인용 도구 / 작은 맥 앱 / Figma Plugin / 실제 실행 / 업무 확장

말할 흐름:
- 개인에게 필요한 작은 도구도 만들 수 있음.
- 예를 들면 마우스 포인터 강조, 드래그 영역 표시 같은 작은 업무 보조 도구.
- Figma Plugin처럼 디자인 도구 안에서 동작하는 확장 기능도 만들 수 있음.
- 핵심은 “내가 반복해서 불편한 일”을 도구로 바꿀 수 있다는 가능성.

다음 연결:
- 꼭 처음부터 직접 만들 필요는 없음. 이미 만들어진 Agent를 참고할 수 있음.

## 25. 당신의 일에 맞는 Agent가 이미 있습니다

키워드: Agent Catalog / 빈 문서에서 시작하지 않기 / 직무별 Agent / 가져와서 수정

말할 흐름:
- Agent는 처음부터 다 직접 만들 필요가 없음.
- 콘텐츠, SEO, UI, UX, 채용처럼 직무별 Agent 예시가 이미 있음.
- 그대로 쓰기보다 우리 팀의 말투, 기준, 업무 방식에 맞게 고쳐 쓰는 출발점으로 보면 좋음.
- 카드를 눌러 실제 Agent 파일을 볼 수 있음.

다음 연결:
- Agent처럼 플러그인과 스킬도 추천 출발점이 있음.

## 26. 추천 플러그인과 주요 스킬

키워드: Superpowers / writing-plans / brainstorming / oh-my-codex / deep-interview / ralph

말할 흐름:
- 처음부터 많은 기능을 다 쓰려고 하지 않아도 됨.
- Superpowers는 계획을 세우고 아이디어를 넓힐 때 좋은 출발점.
- writing-plans는 큰 목표를 실행 가능한 계획으로 나누고, brainstorming은 아이디어를 여러 방향으로 펼침.
- oh-my-codex는 Codex를 더 체계적으로 움직이는 흐름.
- deep-interview는 애매한 요청을 질문으로 좁히고, ralph는 끝까지 실행하고 검증하는 흐름.

다음 연결:
- 이제 전체를 정리하고, 각자 가장 먼저 맡기고 싶은 일을 질문으로 열어둠.

## 27. Q & A

키워드: 질문 / 가장 먼저 맡길 일 / 문서 / 디자인 도구 / 사내 시스템

말할 흐름:
- 오늘의 핵심은 AI를 더 많이 쓰는 것이 아니라, 내 기준을 더 잘 전달하고 저장하는 것.
- 프롬프트에서 시작해서 Instruction, Memory, Skill, Agent로 기준을 쌓고, MCP와 Plugin으로 도구까지 연결할 수 있음.
- 질문이 없으면 “여러분의 문서, 디자인 도구, 사내 시스템과 연결된다면 가장 먼저 어떤 일을 맡기고 싶나요?”로 대화 유도.

마무리 한 줄:
- AI는 답변하는 창에서, 내 기준으로 함께 만드는 작업자가 될 수 있습니다.
