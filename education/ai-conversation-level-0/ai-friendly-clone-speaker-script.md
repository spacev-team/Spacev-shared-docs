# AI와 더 친하게 지내는법

부제: 해야 할 것과 하지 말아야 할 것을 설정하여 나의 클론 만들기

대상: 사내 구성원, AI 입문자, 비개발자 중심

목표: AI를 잘 쓰기 위한 기본 개념을 전달하고, Instruction, Memory, Skill, Agent를 “내 업무 기준을 반복 가능하게 만드는 장치”로 이해하게 한다.

## 01. 표지

오늘의 핵심은 AI를 사람처럼 복제한다는 뜻이 아닙니다.

내가 원하는 기준, 말투, 출력 방식, 검토 방식을 AI가 더 잘 참고하게 만들어 “내 업무 방식의 클론”에 가까워지는 과정이라고 설명합니다.

## 02. 오늘의 흐름

좋은 프롬프트에서 시작해 AI가 참고하는 요소를 봅니다.

그다음 결과가 어긋나는 이유를 사례로 확인하고, Skill과 Agent로 반복 업무를 저장한 뒤, Codex에서 지침과 Memory를 설정하는 방법으로 이어집니다.

## 03. 오프닝

AI와 친해진다는 말을 “내가 원하는 일의 기준을 함께 정하는 것”으로 설명합니다.

오늘은 AI를 완벽한 복제 인간으로 만드는 이야기가 아니라, 내 목적, 판단 기준, 금지선, 일하는 순서를 AI가 더 잘 참고하게 만드는 방법을 다룹니다.

## 04. 퀴즈: 좋은 프롬프트란 무엇일까요?

바로 정답을 말하지 말고 수강생에게 먼저 묻습니다.

“어떤 정보가 있어야 AI가 일을 잘할까요?”라고 질문하고, 나온 답을 다음 장의 기준과 연결합니다.

## 05. 좋은 프롬프트란?

좋은 프롬프트는 긴 문장이 아닙니다.

역할, 목표, 맥락, 해야 할 것 / 하지 말아야 할 것, 출력 형식, 확인 기준이 분리되어 있으면 AI가 무엇을 기준으로 판단해야 하는지 이해하기 쉽습니다.

화면의 왼쪽과 오른쪽은 같은 요청입니다.

둘 다 “비개발자에게 나의 클론 만들기를 요리사 예시로 설명하는 교육자료”를 만들라는 요청입니다.

차이는 오른쪽에는 역할, 대상, 목표, 맥락, 해야 할 것과 하지 말아야 할 것, 출력 형식, 확인 기준이 분리되어 있다는 점입니다.

요리사 예시는 “손님 취향, 금지 재료, 순서, 결과 형태를 확인해야 좋은 결과가 나온다”는 정도로 자연스럽게 연결합니다.

## 06. 내 업무 클론을 이루는 기본 요소

이제 좋은 프롬프트의 항목들이 실제 AI 도구 안에서 어떤 형태로 저장되고 반복되는지 봅니다.

핵심은 기준, 기억, 반복 절차, 역할을 구분하는 것입니다.

## 07. Instruction

Instruction은 총 주방장이 항상 지켜야 하는 운영 원칙입니다.

여기서는 총 주방장이 메인 역할입니다.

그래서 지침에는 “손님 요청을 먼저 확인한다”, “담당 Agent에게 일을 나눈다”, “최종 결과를 검수한다” 같은 기준이 들어갑니다.

해야 할 것은 손님 수, 금지 재료, 선호 맛, 완성 시간을 먼저 확인하는 것입니다.

그리고 재료 손질, 양식, 찜 요리 담당처럼 역할을 나누고, 마지막에는 간, 익힘, 플레이팅을 확인합니다.

하지 말아야 할 것은 손님이 말하지 않은 재료를 마음대로 추가하거나, 담당 Agent의 결과를 검토 없이 그대로 내는 것입니다.

Instruction은 이렇게 매번 반복해서 지켜야 하는 기준을 파일이나 설정으로 남기는 장치라고 설명합니다.

## 08. Memory

Memory는 총 주방장이 계속 참고하면 좋은 반복 기억입니다.

Instruction이 “반드시 지켜야 하는 운영 기준”이라면, Memory는 “참고하면 더 잘 맞춰줄 수 있는 정보”입니다.

예를 들어 단골 손님이 마늘 향은 좋아하지만 너무 매운맛은 싫어한다는 정보, 쉬운 말로 설명하면 더 잘 이해한다는 정보, 결과를 정리할 때 요리 이름과 맛의 특징을 먼저 보여주는 방식을 좋아한다는 정보가 Memory에 들어갈 수 있습니다.

하지만 알레르기나 금지 재료처럼 반드시 지켜야 하는 조건은 Memory에만 기대면 위험합니다.

그래서 중요한 조건은 매번 요청에서 다시 확인해야 한다고 설명합니다.

## 09. Context

Context는 AI와 주고받은 질문, 답변, 수정 요청, 첨부 자료가 이어진 “대화의 맥락”입니다.

수강생에게는 “AI에게 질문하고, 답을 받고, 다시 고쳐달라고 말하는 그 흐름 전체”라고 설명합니다.

예를 들어 “비개발자 대상”, “쉬운 말”, “화면에 비유라는 단어는 넣지 않기” 같은 조건이 대화 맥락 안에 있어야 다음 답이 맞춰집니다.

## 10. Context Window

Context Window는 AI가 한 번에 참고할 수 있는 정보의 범위입니다.

화면의 막대 전체가 참고 범위이고, 요청, 지침, Memory, 이전 대화가 그 안에서 공간을 나눠 가진다고 설명합니다.

대화가 길어지면 이전 내용은 압축되고, 세부 표현이나 작은 조건 같은 데이터 일부는 유실될 수 있습니다.

중요한 기준을 새 요청에 다시 적으면 막대 안에서 그 기준의 비중이 커집니다.

## 11. Skill

Skill은 반복 업무 절차입니다.

특정 요리를 만들 때 재료 확인, 도구 준비, 조리 순서, 마지막 맛 확인이 정해져 있는 것처럼 반복되는 순서를 저장합니다.

토마토 파스타 Skill 예시처럼 순서를 파일로 남겨두면 짧은 요청만 해도 같은 조리 방식이 반복됩니다.

## 12. Agent

Agent는 역할을 맡은 AI입니다.

화면의 예시는 총 주방장 Agent입니다.

손님 요청과 금지 재료를 확인하고, 재료 손질 담당 Agent, 양식 담당 Agent, 찜 요리 담당 Agent에게 일을 나눈 뒤 결과를 모아 최종 코스를 완성합니다.

Skill이 절차라면 Agent는 책임과 판단 기준을 가진 역할이라고 설명합니다.

## 13. 개념 한 줄 정리

표를 천천히 읽습니다.

Prompt는 이번 요청, Instruction은 항상 지킬 기준, Context는 이번 작업의 참고 자료, Context Window는 참고 범위, Memory는 개인 선호, Skill은 반복 절차, Agent는 역할입니다.

## 14. 불편했던 사례로 다시 보기

이제 개념을 실제 불편함에 연결합니다.

AI가 이상하게 군다고 느끼는 순간은 대개 요청, 기준, 기억, 확인 방식 중 하나가 흐려졌기 때문입니다.

## 15. 하지 말라고 했는데 자꾸 말을 안 들어요

왼쪽 대화는 “하지 말라”고 했는데도 AI가 금지 표현을 다시 쓰는 장면입니다.

오른쪽은 왜 이런 일이 생기는지 정리한 진단입니다.

현상은 금지한 표현이 다시 나오고, 수정할수록 기준이 흐려지는 것입니다.

원리는 Instruction, Prompt, Context가 함께 작동한다는 점입니다.

계속 지킬 금지선은 Instruction에 있어야 하고, 이번 작업에서 강하게 반영할 내용은 Prompt에도 다시 들어가야 합니다.

고쳐야 할 부분은 반복 금지선을 Instruction에 저장하고, 중요한 조건은 새 요청 맨 앞에 다시 쓰는 것입니다.

## 16. 분명히 말했는데 또 까먹었어요

왼쪽 대화는 “비개발자 대상”이라고 말했는데도 어려운 말이 다시 나오는 장면입니다.

오른쪽에서는 현상, 원리와 개념, 고쳐야 할 부분을 순서대로 봅니다.

현상은 앞에서 정한 대상과 톤이 약해지는 것입니다.

원리는 Context, Context Window, Memory가 연결됩니다.

AI는 현재 창 안에 강하게 들어온 정보를 중심으로 답합니다.

긴 대화는 요약되며 힘이 약해지고, Memory도 참고용이므로 필수 조건은 이번 요청에 다시 적어야 합니다.

그래서 “대상은 비개발자, 쉬운 예시 먼저”처럼 중요한 기준을 새 요청 맨 앞에 다시 적는 것이 좋습니다.

## 17. 틀렸는데 너무 자신 있게 말해요

왼쪽 대화는 AI가 확인되지 않은 최신 정보를 단정하는 장면입니다.

현상은 근거가 없는데도 자연스럽고 자신 있게 말한다는 것입니다.

원리는 Context, Instruction, Skill이 연결됩니다.

근거 자료가 Context에 없으면 AI는 그럴듯한 추측을 만들 수 있습니다.

그래서 “모르면 모른다고 말해”라는 Instruction과, 근거 확인 체크리스트를 수행하는 Skill이 필요합니다.

고쳐야 할 부분은 근거와 추측을 분리하게 하고, 마지막에 확인 체크리스트를 실행하게 하는 것입니다.

## 18. 역할을 나눠 일하는 AI

Main Agent는 전체 요청과 기준을 보고, 필요한 Skill과 Sub Agent를 선택해 결과물을 완성합니다.

기술적으로 깊게 들어가기보다 “전체를 보는 역할과 맡아서 처리하는 역할이 나뉜다”는 정도로 설명합니다.

## 19. 5컷으로 보는 AI 작업 흐름

흐름은 요청 이해, 참고 자료 확인, 절차와 역할 선택, 위임, 최종 정리입니다.

각 단계가 앞에서 배운 Prompt, Context, Skill, Agent와 연결된다고 짚어줍니다.

## 20. 요청과 참고 자료를 확인한다

AI는 이번 요청만 보는 것이 아니라 지침, 기억, 파일, 이전 대화도 함께 참고하려고 합니다.

현재 참고 범위 안에 있는 정보가 이번 결과에 더 강하게 영향을 줍니다.

## 21. 절차와 역할을 고르고 일을 맡긴다

Skill은 절차이고 Sub Agent는 역할입니다.

어떤 순서로 만들지와 누가 어떤 관점으로 볼지를 분리하면 반복 작업이 더 안정됩니다.

## 22. 결과를 검토하고 전달한다

Sub Agent가 만든 결과는 초안입니다.

Main Agent가 원래 요청, 기준, 대상, 출력 형식을 다시 확인하고 최종 결과로 정리합니다.

## 23. 내 업무 방식을 저장하기

이제 내 업무 기준을 어떻게 저장할지로 넘어갑니다.

목표는 매번 긴 프롬프트를 쓰는 것이 아니라 기준, 절차, 역할표를 저장하는 것입니다.

## 24. Skill은 반복 업무 절차입니다

실제 SKILL.md처럼 번호가 붙은 절차를 봅니다.

토마토 파스타 예시처럼 주문 확인, 도구 준비, 재료 준비, 조리 순서, 최종 확인을 파일로 남기면 짧은 요청만 해도 같은 흐름을 다시 실행할 수 있습니다.

여기서 핵심은 “내가 선호하는 순서”를 저장하는 것입니다.

## 25. Agent는 역할을 맡은 AI입니다

Agent의 핵심은 특정 역할을 배정하는 것입니다.

화면의 .md 예시는 실제 요리 담당 역할처럼 작성되어 있습니다.

재료 손질 담당 Agent는 조리 전 재료 준비, 양식 담당 Agent는 양식 메뉴 조리, 찜 요리 담당 Agent는 찜기의 순서와 익힘 시간을 맡습니다.

이처럼 역할, 해야 할 일, 결과물이 서로 맞아야 Main Agent가 정확히 위임할 수 있다고 설명합니다.

## 26. 클론은 한 번에 완성되지 않습니다

처음부터 완벽한 기준을 만들 필요는 없습니다.

너무 어렵다, 근거가 부족하다, 예시가 약하다 같은 피드백을 Instruction, Memory, Skill, Agent로 옮기면 다음 첫 결과가 조금씩 좋아집니다.

## 27. 오늘부터 만드는 내 AI 클론 체크리스트

수강생이 오늘 바로 할 수 있는 네 가지를 정리합니다.

해야 할 것 / 하지 말아야 할 것 쓰기, 운영 원칙 저장하기, 절차를 Skill화하기, 역할 나누기입니다.

## 28. 핵심 3줄

세 문장을 천천히 읽습니다.

AI와 친해지는 첫걸음은 해야 할 것과 하지 말아야 할 것이고, Instruction, Memory, Skill, Agent는 그 기준을 반복 가능하게 만드는 장치입니다.

## 29. 업무 방식의 클론

클론을 과장하지 않게 정리합니다.

완벽한 복제본이 아니라, AI가 내 기준과 절차를 참고해 처음부터 조금 더 나답게 일하도록 만드는 과정입니다.

## 30. Codex 사용 전: 개인 맞춤 지침 설정

스크린샷의 번호를 따라 설정 위치를 설명합니다.

1번은 왼쪽 메뉴의 개인 맞춤 설정, 2번은 맞춤형 지침 입력칸, 3번은 내 정보 입력 영역입니다.

맞춤형 지침에는 해야 할 것, 하지 말아야 할 것, 말투, 출력 형식을 적고, 내 정보에는 내 역할과 자주 하는 일을 적습니다.

## 31. Memory: 반복 선호를 저장하기

Memory는 매번 다시 말하지 않아도 좋은 선호를 저장하는 기능입니다.

화면의 번호를 따라 메모리 영역, 관리 버튼, 저장된 메모리 참고 설정을 확인합니다.

Memory는 설정 화면의 텍스트 입력칸에 직접 쓰는 방식이 아니라, 대화창에서 “기억해줘: 나는 쉬운 예시와 짧은 화면 문구를 선호해”처럼 말해 저장합니다.

다만 필수 조건은 Memory에만 맡기지 말고 이번 요청에도 다시 적습니다.

## 32. Codex와 Claude Code로 할 수 있는 일

Codex와 Claude Code 같은 제작형 AI는 대화만 하는 것이 아니라 문서, 작은 웹 결과물, 자동화, 검토와 개선까지 함께할 수 있습니다.

“AI는 답변하는 창에서 내 기준으로 함께 만드는 작업자로 확장된다”는 말로 가능성을 부드럽게 정리합니다.

## 33. Q&A

질문을 받습니다.

질문이 없으면 “내가 자주 반복해서 시키는 일 하나를 떠올려보세요. 그 일의 해야 할 것, 하지 말아야 할 것, 반복 절차, 필요한 역할은 무엇일까요?”라고 유도합니다.
