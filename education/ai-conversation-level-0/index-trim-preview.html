PART 1

AI를 이루는 기본 요소

먼저 볼 개념들

뒤에서 하나씩 살펴볼 핵심 개념을 먼저 짧게 정리합니다
AI가 내 기준대로 일하려면, 기준·기억·절차·역할·대화 범위를 나눠서 이해하면 쉽습니다. 지금은 이름과 역할만 먼저 보고, 다음 페이지부터 하나씩 예시로 살펴봅니다.

Instruction

AI가 항상 따라야 할 역할과 기준입니다.

Memory

반복해서 설명하기 어려운 취향과 기준을 저장합니다.

Skill

반복 작업의 순서와 체크리스트를 저장합니다.

Agent

특정 역할과 책임을 맡아 판단하고 수행합니다.

Context

질문과 답변이 이어진 현재 대화의 흐름입니다.

Context Window

AI가 한 번에 참고할 수 있는 정보의 범위입니다.

Instruction(지침)

AI가 답변할 때 항상 따라야 할 역할, 행동 기준, 하지 말아야 할 것, 참고 정보를 적습니다

항상 지켜야 하는 기준

역할해야 할 일피해야 할 일참고 정보

언제 쓰나

AI가 매번 같은 역할과 기준으로 답변해야 할 때.

AI가 보는 방식

AI가 사용자의 질문에 대한 분석을 진행하기 전에, 항상 가장 먼저 읽는 것이 지침입니다. 이 지침은 “역할”, “해야 할 것과 하지 말아야 할 것” 등의 내용을 포함합니다.

# AGENTS.md 또는 CLAUDE.md 예시 # 총 주방장 운영 지침 ## 역할 - 전체 기준을 정하고 담당 Agent에게 일을 나눈다. ## 해야 할 것 - 주문 조건과 금지 재료를 먼저 확인한다. - 담당 Agent에게 역할과 결과물 형식을 전달한다. - 최종 결과의 간, 익힘, 플레이팅을 점검한다. ## 하지 말아야 할 것 - 손님이 말하지 않은 재료를 임의로 추가하지 않는다. - 담당 Agent 결과를 확인 없이 내지 않는다.

Memory(기억)

총 주방장이 계속 참고하면 좋은 손님 취향과 반복 정보를 남깁니다

반복 기억

취향반복 기준선호 표현자주 쓰는 예시

언제 쓰나

매번 새 대화를 열 때마다 똑같은 설명을 하지 않기 위해 사용합니다.

지침과 차이

Instruction은 질문을 분석하기 전에 항상 먼저 읽는 기준이고, Memory는 새 대화마다 반복 설명하지 않기 위해 참고하는 기억입니다.

# memory.md 예시 # 총 주방장이 참고할 반복 기억 - 단골 손님은 마늘 향은 좋아하지만 너무 맵게 먹지 않는다. - 설명할 때는 어려운 조리 용어보다 쉬운 말로 먼저 안내한다. - 결과를 낼 때는 “요리 이름, 맛의 특징, 확인한 기준” 순서로 정리한다. - 예시는 토마토 파스타처럼 익숙한 요리로 들면 이해가 빠르다. - 단, 알레르기와 금지 재료는 Memory에만 맡기지 말고 매번 요청에서 다시 확인한다.

Skill(업무 절차)

SKILL.md에 반복되는 업무 순서와 검토 기준을 저장합니다

반복 실행 절차

시작 조건진행 순서확인 기준결과 형식

언제 쓰나

매번 비슷한 순서로 처리하는 일이 있을 때 사용합니다. 예: 자료 정리, 회의록 요약, 안내문 작성, 검토 체크리스트.

좋은 점

요청을 짧게 해도 정해둔 순서, 확인 기준, 결과 형식이 반복되어 결과 품질이 흔들리지 않습니다.

# SKILL.md 예시 --- name: tomato-pasta description: 내가 선호하는 순서로 토마토 파스타 만들기 --- 1. 손님 수, 금지 재료, 완성 시간을 확인한다. 2. 냄비, 팬, 집게, 체, 접시를 준비한다. 3. 물을 끓이고 면을 삶는다. 4. 팬에 마늘, 토마토소스, 면수를 넣어 소스를 만든다. 5. 면을 소스에 버무리고 간과 플레이팅을 확인한다.

Agent(역할을 맡은 AI)

AGENT.md에 역할, 위임 기준, 결과 확인 방식을 적습니다

역할 담당자

담당 역할판단 기준위임 범위결과 책임

언제 쓰나

한 사람이 다 하기보다 역할을 나누면 더 좋은 일에 사용합니다. 예: 자료 조사 담당, 초안 작성 담당, 검토 담당.

Skill과 차이

Skill은 “어떤 순서로 할지”를 정하고, Agent는 “누가 어떤 관점으로 판단하고 책임질지”를 정합니다.

# AGENT.md 예시 # 총 주방장 Agent ## 역할 - 손님 요청과 금지 재료를 확인한다. - 담당 Agent에게 일을 나누고 결과를 모은다. - 최종 코스의 맛, 순서, 완성 시간을 책임진다. ## 위임 기준 - 재료 준비 → 재료 손질 담당 Agent - 파스타/스테이크 → 양식 담당 Agent - 찜 요리 → 찜 요리 담당 Agent ## 진행 순서 1. 손님 취향, 알레르기, 완성 시간을 확인한다. 2. 담당 Agent에게 역할과 결과물 형식을 전달한다. 3. 중간 결과를 점검하고 수정 요청을 보낸다. 4. 최종 메뉴 구성과 플레이팅을 확인한다. ## 하지 말아야 할 것 - 담당 Agent 결과를 확인 없이 내지 않는다. - 금지 재료나 알레르기 정보를 무시하지 않는다.

Context(대화의 흐름)

AI와 주고받은 질문과 답, 수정 요청, 첨부 자료가 이번 답변의 배경이 됩니다
나비개발자에게 AI 기초를 쉽게 설명하는 자료를 만들고 싶어.
AI좋아요. 먼저 좋은 프롬프트, 기억, 지침, 역할을 나눠 설명해볼게요.
나요리사 예시를 쓰되, “비유”라는 단어는 화면에 넣지 말아줘.
AI그러면 요리 흐름은 자연스럽게 녹이고, 화면 문구는 교육 주제 중심으로 정리하겠습니다.

Context란?

지금 대화에서 오간 말의 흐름입니다. AI는 방금 전 질문과 답변, 수정 요청을 함께 보고 다음 답을 만듭니다.

왜 중요할까?

AI를 쓰며 생기는 많은 이슈는 필요한 맥락이 빠졌거나 오래된 맥락을 계속 붙잡을 때 생깁니다.

  • 대상을 잊고 너무 어렵게 설명함
  • 방금 바꾼 기준 대신 이전 기준으로 답함
  • 첨부 자료의 목적을 다르게 이해함

실무 팁

대화가 길어지면 중요한 조건을 새 요청 첫 줄에 다시 적습니다. 예: “조건 유지: 대상은 고객, 3문장 이하, 다음 행동을 마지막에 포함.”

Context Window(참고 범위)

대화와 자료가 길어지면 모든 내용이 그대로 남지 않고 일부가 빠질 수 있습니다
참고 범위

Context Window는 AI가 한 번에 참고할 수 있는 정보의 범위입니다. 이 범위를 넘으면 대화는 요약되고, 세부 내용 일부는 빠질 수 있습니다.

1. 대화가 쌓임요청과 답변이 계속 이어져요

대화를 이어갈수록 이전 요청, 답변, 수정 사항, 참고 자료가 Context Window 안에 쌓입니다.

범위를 넘으면 모든 내용을 그대로 들고 가지 못해서, 이전 대화가 짧은 요약본으로 바뀝니다.

요약 이후에도 대화는 계속 이어지지만, 빠진 세부 조건은 AI가 놓칠 수 있습니다.

실무 팁
계속 유지되어야 하는 기준은 대화에만 남기지 말고 Memory에 저장합니다.
예: “앞으로 고객 안내문은 정중하고 짧게, 핵심 일정과 다음 행동을 꼭 포함해줘.”

처음부터 완벽한 결과물이 만들어지지는 않습니다.

반복적인 피드백을 통해 점점 더 나은 결과물을 만들어낼 수 있습니다.
1

“너무 어렵다”, “근거가 부족하다”, “예시가 약하다”처럼 기준을 줍니다.

PART 2

불편했던 사례로 다시 보기

“하지 말라고 했는데 자꾸 말을 안 들어요.”

한 번의 요청과 계속 지켜야 하는 지침을 구분해야 합니다
나고객에게 보낼 배송 지연 안내문이야. 변명처럼 들리는 말은 쓰지 마.
AI물류 상황상 불가피하게 지연되었으니 양해 부탁드립니다...
나변명처럼 쓰지 말라고 했잖아.
AI죄송합니다. 금지 표현을 빼고 다시 작성하겠습니다.
이상한 답변을 왜 할까요?

현상

피하고 싶은 표현이 다시 나오고, 수정할수록 말투 기준이 흐려집니다.

문제와 설정 위치

  • 계속 피해야 하는 표현 → Instruction: “변명처럼 들리는 표현은 쓰지 마.”
  • 이번 답변에서 중요 → Prompt: “배송 지연 안내문이고, 변명처럼 들리지 않게 써줘.”
  • 마지막 확인 필요 → 검수 요청: “변명처럼 들리는 표현이 있는지 점검해줘.”

계속 피해야 하는 표현은 저장하고, 이번 답변에서 중요한 조건은 요청 앞에 다시 적습니다.

고쳐야 할 부분

  • 계속 피해야 하는 표현은 Instruction에 저장합니다.
  • 중요한 조건은 이번 요청 맨 앞에 다시 적습니다.
  • 마지막에 “변명처럼 들리는 표현이 있는지 확인해줘”를 붙입니다.

AI안녕하세요 고객님. 먼저 이용에 불편을 드려 죄송합니다. 현재 물류센터 사정으로 인해...

이상한 답변을 왜 할까요?

현상

앞에서 정한 길이와 말투 조건이 약해지고, 긴 대화일수록 조건이 섞입니다.

문제와 설정 위치

  • 3문장 조건을 잊음 → Prompt: “3문장 이하로 작성해줘.”
  • 정중한 말투가 반복 필요 → Memory: “고객 안내문은 항상 정중하고 짧게 작성해줘.”
  • 대화가 길어짐 → 새 요청: “조건 유지: 3문장 이하, 정중한 말투.”

계속 유지할 기준은 Memory에 저장하고, 이번 답변의 필수 조건은 Prompt에 다시 적습니다.

고쳐야 할 부분

  • 중요한 기준은 새 요청 맨 앞에 다시 적습니다.
  • “3문장 이하, 정중한 말투”처럼 짧게 고정합니다.
  • 반복 선호는 Memory에, 필수 조건은 요청에 넣습니다.

현상

근거가 없는데도 확실한 것처럼, 최신 정보나 숫자를 그럴듯하게 말합니다.

문제와 설정 위치

  • 근거 없이 확실한 것처럼 말함 → Instruction: “확인 못 한 내용은 추측이라고 표시해.”
  • 최신 정보가 필요함 → Prompt: “최신 정보인지 확인하고, 모르면 모른다고 말해줘.”
  • 반복 확인 필요 → Skill: “근거 확인 → 추측 분리 → 체크리스트 순서로 검토해.”

확인 기준은 Instruction에 저장하고, 반복 검증은 Skill로 절차화합니다.

고쳐야 할 부분

  • “확인 못 한 내용은 추측이라고 표시해”를 지침에 넣습니다.
  • 근거와 추측을 분리해서 쓰게 합니다.
  • 마지막에 확인 체크리스트를 실행하게 합니다.

Codex와 Claude Code

AI가 외부 도구와 연결되고, 실제 업무 결과물을 만들어내는 방식을 살펴봅니다.

MCP란?

Notion, Figma, Slack처럼 서로 다른 서비스도 정해진 연결 규칙으로 AI가 다룰 수 있습니다

AI와 외부 도구를
정해진 규칙으로 연결합니다

MCP

MCP(Model Context Protocol)는 AI가 외부 서비스의 데이터와 기능을 정해진 방식으로 읽고 실행하게 해주는 연결 표준입니다. 사용자의 명령어가 MCP를 지나 실제 산출물로 바뀝니다.

PlugIn 이란?

PLUGIN

AI에 기능을 꽂아 쓰는 연결 묶음입니다

AI 자체를 바꾸는 것이 아니라, 필요한 도구와 절차를 플러그처럼 연결해 더 많은 일을 맡기는 방식입니다.

MCPNotion, Figma, Slack 같은 외부 서비스와 연결합니다.
Skill반복되는 업무 절차와 검토 기준을 불러옵니다.
Agent자료 정리, 검토, 제작처럼 역할을 나눠 맡깁니다.
Tool문서, 파일, 브라우저 같은 실제 작업 기능을 실행합니다.

핵심: AI에게 기능 플러그를 꽂아, 답변을 넘어 실제 작업까지 맡깁니다.
