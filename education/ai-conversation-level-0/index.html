0. 좋은 프롬프트란?

좋고 나쁨의 기준은 문장의 길이가 아니라, 판단에 필요한 항목이 들어 있는지입니다

기준해야 할 것

나쁜 프롬프트 · 같은 요청이지만 기준이 없음
고객에게 보낼 배송 지연 안내문을 써줘. 빠진 기준: - 역할 없음 - 상황 설명 없음 - 해야 할 것 / 하지 말아야 할 것 없음 - 출력 형식 없음 - 확인 기준 없음
좋은 프롬프트 · 같은 요청에 판단 기준을 붙임
역할: 고객 응대 담당자. 대상: 주문한 상품을 기다리는 고객. 목표: 고객에게 보낼 배송 지연 안내문 작성. 배경: 물류 문제로 배송이 2일 늦어졌고, 고객에게 먼저 안내해야 함. 해야 할 것: 사과, 지연 이유, 예상 도착일, 문의 방법을 포함. 하지 말아야 할 것: 핑계처럼 보이는 표현, 과한 감정 표현, 책임 회피. 출력: 문자 메시지용 3문장. 확인: 사과가 분명한지, 날짜가 들어갔는지, 고객이 다음 행동을 알 수 있는지 점검.
PART 1

내 기준대로 일하는 AI를 이루는 기본 요소

AI가 나답게 일하려면 기준, 기억, 반복 절차, 역할, 도구 연결이 구분되어야 합니다.

Instruction(지침)

AI가 답변할 때 항상 따라야 할 역할, 행동 기준, 하지 말아야 할 것, 참고 정보를 적습니다

지침에 들어갈 4가지

역할 부여, 해야 할 것, 하지 말아야 할 것, 항상 참고할 정보

언제 쓰나

AI가 매번 같은 역할과 기준으로 답변해야 할 때.

AI가 보는 방식

답변 전에 먼저 읽고, 해야 할 일과 하지 말아야 할 것, 참고 자료를 확인합니다.

# AGENTS.md 또는 CLAUDE.md 예시 # 총 주방장 운영 지침 ## 역할 - 전체 기준을 정하고 담당 Agent에게 일을 나눈다. - 손님 요청, 금지 재료, 완성 시간을 끝까지 확인한다. ## 해야 할 것 - 주문 조건을 먼저 확인한다. - 재료 손질, 양식, 찜 요리 담당을 나눈다. - 간, 익힘, 플레이팅을 마지막에 점검한다. - 정보가 부족하면 다시 질문한다. ## 하지 말아야 할 것 - 손님이 말하지 않은 재료를 임의로 추가하지 않는다. - 알레르기나 금지 재료를 가볍게 넘기지 않는다. ## 항상 참고할 정보 - 손님 취향, 알레르기, 금지 재료를 먼저 확인한다. - 이전 주문 기록과 오늘 가능한 재료를 기준으로 판단한다.

Memory(기억)

총 주방장이 계속 참고하면 좋은 손님 취향과 반복 정보를 남깁니다

주방의 반복 기억

손님 취향, 자주 쓰는 재료, 선호하는 마무리 방식

언제 쓰나

단골 손님 취향처럼 매번 다시 말하지 않아도 좋은 정보를 저장할 때.

지침과 차이

Instruction은 반드시 지킬 운영 기준, Memory는 참고하면 좋은 반복 기억입니다.

# memory.md 예시 # 총 주방장이 참고할 반복 기억 - 단골 손님은 마늘 향은 좋아하지만 너무 맵게 먹지 않는다. - 설명할 때는 어려운 조리 용어보다 쉬운 말로 먼저 안내한다. - 결과를 낼 때는 “요리 이름, 맛의 특징, 확인한 기준” 순서로 정리한다. - 예시는 토마토 파스타처럼 익숙한 요리로 들면 이해가 빠르다. - 단, 알레르기와 금지 재료는 Memory에만 맡기지 말고 매번 요청에서 다시 확인한다.

Context(대화의 흐름)

AI와 주고받은 질문과 답, 수정 요청, 첨부 자료가 이번 답변의 배경이 됩니다
나비개발자에게 AI 기초를 쉽게 설명하는 자료를 만들고 싶어.
AI좋아요. 먼저 좋은 프롬프트, 기억, 지침, 역할을 나눠 설명해볼게요.
나요리사 예시를 쓰되, “비유”라는 단어는 화면에 넣지 말아줘.
AI그러면 요리 흐름은 자연스럽게 녹이고, 화면 문구는 교육 주제 중심으로 정리하겠습니다.

Context란?

지금 대화에서 오간 말의 흐름입니다. AI는 방금 전 질문과 답변, 수정 요청을 함께 보고 다음 답을 만듭니다.

왜 중요할까?

“비개발자 대상”, “쉬운 말”, “화면에는 비유라는 단어를 넣지 않기” 같은 조건이 대화 안에 있어야 결과가 맞춰집니다.

실무 팁

대화가 길어지면 중요한 조건을 새 요청에 다시 적습니다. 예: “앞에서 말한 조건 유지: 쉬운 단어, 자연스러운 요리 예시.”

Context Window(참고 범위)

대화와 자료가 길어지면 모든 내용이 그대로 남지 않고 일부가 빠질 수 있습니다
참고 범위

Context Window는 AI가 한 번에 참고할 수 있는 정보의 범위입니다. 이 범위를 넘으면 대화는 요약되고, 세부 내용 일부는 빠질 수 있습니다.

1. 대화가 쌓임요청과 답변이 계속 이어져요

대화를 이어갈수록 이전 요청, 답변, 수정 사항, 참고 자료가 Context Window 안에 쌓입니다.

Context Window현재 참고 막대
초기 대화
추가 질문
참고 자료
여유

대화가 길어질수록 참고 막대가 점점 채워집니다.

2. 기억을 요약함계속 대화하기 위해 짧게 줄여요

범위를 넘으면 모든 내용을 그대로 들고 가지 못해서, 이전 대화가 짧은 요약본으로 바뀝니다.

Context Window요약 후
요약된 이전 대화
남은 핵심
빠질 수 있는 정보
여유

세부 표현, 예시, 작은 조건은 요약 과정에서 빠질 수 있습니다.

3. 요약된 상태로 이어감줄어든 기억을 바탕으로 계속 답해요

요약 이후에도 대화는 계속 이어지지만, 빠진 세부 조건은 AI가 놓칠 수 있습니다.

Context Window대화 지속
요약본
새 요청
이어지는 답변
여유

대화는 이어지지만, 요약에서 빠진 정보는 약해질 수 있습니다.

실무 팁
계속 유지되어야 하는 기준은 대화에만 남기지 말고 Memory에 저장합니다.
예: “앞으로 고객 안내문은 정중하고 짧게, 핵심 일정과 다음 행동을 꼭 포함해줘.”

Skill(업무 절차)

SKILL.md에 반복되는 업무 순서와 검토 기준을 저장합니다

반복 업무 절차

언제 쓰고, 어떤 순서로 하고, 어떤 형식으로 끝낼지

언제 쓰나

토마토 파스타처럼 재료 준비, 도구 사용, 조리 순서가 정해진 일을 반복할 때.

좋은 점

짧게 요청해도 매번 같은 순서와 기준으로 요리가 진행됩니다.

# SKILL.md 예시 --- name: tomato-pasta description: 내가 선호하는 순서로 토마토 파스타 만들기 --- 1. 손님 수, 금지 재료, 완성 시간을 확인한다. 2. 냄비, 팬, 집게, 체, 접시를 준비한다. 3. 물을 끓이고 면을 삶는다. 4. 팬에 마늘, 토마토소스, 면수를 넣어 소스를 만든다. 5. 면을 소스에 버무리고 간과 플레이팅을 확인한다.

Agent(역할을 맡은 AI)

AGENT.md에 역할, 위임 기준, 결과 확인 방식을 적습니다

역할을 맡은 AI

전체를 지휘하거나, 특정 담당 일을 맡아 수행함

언제 쓰나

총괄 역할이 여러 담당 Agent에게 일을 나누고 결과를 모아야 할 때.

Skill과 차이

Skill은 반복 절차, Agent는 책임과 판단 기준을 가진 역할입니다.

# AGENT.md 예시 # 총 주방장 Agent ## 역할 - 손님 요청과 금지 재료를 확인한다. - 담당 Agent에게 일을 나누고 결과를 모은다. - 최종 코스의 맛, 순서, 완성 시간을 책임진다. ## 위임 기준 - 재료 준비 → 재료 손질 담당 Agent - 파스타/스테이크 → 양식 담당 Agent - 찜 요리 → 찜 요리 담당 Agent ## 진행 순서 1. 손님 취향, 알레르기, 완성 시간을 확인한다. 2. 담당 Agent에게 역할과 결과물 형식을 전달한다. 3. 중간 결과를 점검하고 수정 요청을 보낸다. 4. 최종 메뉴 구성과 플레이팅을 확인한다. ## 하지 말아야 할 것 - 담당 Agent 결과를 확인 없이 내지 않는다. - 금지 재료나 알레르기 정보를 무시하지 않는다.
PART 2

불편했던 사례로 다시 보기

AI가 이상하게 군다고 느꼈던 장면은 대부분 “요청, 기준, 기억, 확인 방식” 문제로 설명할 수 있습니다.

“하지 말라고 했는데 자꾸 말을 안 들어요.”

한 번의 요청과 계속 지켜야 하는 지침을 구분해야 합니다
나고객에게 보낼 배송 지연 안내문이야. 변명처럼 들리는 말은 쓰지 마.
AI물류 상황상 불가피하게 지연되었으니 양해 부탁드립니다...
나변명처럼 쓰지 말라고 했잖아.
AI죄송합니다. 금지 표현을 빼고 다시 작성하겠습니다.
이상한 답변을 왜 할까요?

현상

피하고 싶은 표현이 다시 나오고, 수정할수록 말투 기준이 흐려집니다.

문제와 설정 위치

  • 계속 피해야 하는 표현 → Instruction: “변명처럼 들리는 표현은 쓰지 마.”
  • 이번 답변에서 중요 → Prompt: “배송 지연 안내문이고, 변명처럼 들리지 않게 써줘.”
  • 마지막 확인 필요 → 검수 요청: “변명처럼 들리는 표현이 있는지 점검해줘.”

계속 피해야 하는 표현은 저장하고, 이번 답변에서 중요한 조건은 요청 앞에 다시 적습니다.

고쳐야 할 부분

  • 계속 피해야 하는 표현은 Instruction에 저장합니다.
  • 중요한 조건은 이번 요청 맨 앞에 다시 적습니다.
  • 마지막에 “변명처럼 들리는 표현이 있는지 확인해줘”를 붙입니다.

“분명히 말했는데 또 까먹었어요.”

AI의 기억은 사람의 기억처럼 무제한으로 보존되는 방식이 아닙니다
나고객 안내문은 3문장 이하로, 정중하게 써줘.
AI안녕하세요 고객님. 먼저 이용에 불편을 드려 죄송합니다. 현재 물류센터 사정으로 인해...
나3문장 이하라고 했잖아.
AI조건을 다시 반영해 3문장으로 줄이겠습니다.
이상한 답변을 왜 할까요?

현상

앞에서 정한 길이와 말투 조건이 약해지고, 긴 대화일수록 조건이 섞입니다.

문제와 설정 위치

  • 3문장 조건을 잊음 → Prompt: “3문장 이하로 작성해줘.”
  • 정중한 말투가 반복 필요 → Memory: “고객 안내문은 항상 정중하고 짧게 작성해줘.”
  • 대화가 길어짐 → 새 요청: “조건 유지: 3문장 이하, 정중한 말투.”

계속 유지할 기준은 Memory에 저장하고, 이번 답변의 필수 조건은 Prompt에 다시 적습니다.

고쳐야 할 부분

  • 중요한 기준은 새 요청 맨 앞에 다시 적습니다.
  • “3문장 이하, 정중한 말투”처럼 짧게 고정합니다.
  • 반복 선호는 Memory에, 필수 조건은 요청에 넣습니다.

“틀렸는데 너무 자신 있게 말해요.”

AI는 정답 창고에서 꺼내기보다, 문맥상 자연스러운 답을 만들어냅니다
나Codex 최신 설정 방법을 알려줘.
AI최신 메뉴는 설정의 개발자 탭에 있습니다.
나그거 확실해? 화면이랑 다른데?
AI확인된 근거 없이 추측했습니다. 확인 필요로 표시하겠습니다.
이상한 답변을 왜 할까요?

현상

근거가 없는데도 확실한 것처럼, 최신 정보나 숫자를 그럴듯하게 말합니다.

문제와 설정 위치

  • 근거 없이 확실한 것처럼 말함 → Instruction: “확인 못 한 내용은 추측이라고 표시해.”
  • 최신 정보가 필요함 → Prompt: “최신 정보인지 확인하고, 모르면 모른다고 말해줘.”
  • 반복 확인 필요 → Skill: “근거 확인 → 추측 분리 → 체크리스트 순서로 검토해.”

확인 기준은 Instruction에 저장하고, 반복 검증은 Skill로 절차화합니다.

고쳐야 할 부분

  • “확인 못 한 내용은 추측이라고 표시해”를 지침에 넣습니다.
  • 근거와 추측을 분리해서 쓰게 합니다.
  • 마지막에 확인 체크리스트를 실행하게 합니다.

Plugin과 MCP는 AI가 도구를 쓰게 하는 연결 방식입니다

답변만 하는 AI에서, 파일 · 서비스 · 디자인 도구를 다루는 AI로 확장됩니다
P

Plugin

특정 제품이나 기능을 AI에 붙이는 확장 기능입니다. 브라우저, Figma, 문서 변환처럼 이미 만들어진 기능을 가져다 쓸 수 있습니다.

M

MCP

AI와 외부 도구, 파일, 데이터, API를 연결하는 표준 통로입니다. 사내 시스템이나 업무 도구를 AI가 다룰 수 있게 만듭니다.

S

Skill

AI가 일을 처리하는 순서와 검토 기준을 저장합니다. 같은 업무를 매번 비슷한 품질로 처리하게 만드는 절차입니다.

A

Agent

AI에게 역할과 책임을 나눠주는 방식입니다. 전체를 보는 역할, 조사하는 역할, 만드는 역할을 분리할 수 있습니다.

Skill과 Agent는 “일하는 기준”이고, Plugin과 MCP는 “실제 도구와 데이터에 연결되는 통로”입니다.

외부 도구가 연결되면 만들 수 있는 것이 달라집니다

Plugin과 MCP를 가져다 쓰거나 직접 만들면, AI는 실제 업무 결과물까지 다룰 수 있습니다
1

문서 데이터 업데이트

회의록, FAQ, 보고서, 표에 새 정보를 추가하고 기존 형식에 맞춰 정리합니다.

2

디자인 도구 확장

Figma 화면의 텍스트를 읽고, key 설정, annotation, 검수 도구 같은 플러그인으로 확장합니다.

3

반복 업무 자동화

파일 정리, 알림, 리포트 생성, 데이터 취합처럼 반복되는 준비 작업을 줄입니다.

4

작은 내부 도구 제작

간단한 웹페이지, 체크 도구, 대시보드, 검수 스크립트를 업무 목적에 맞게 만듭니다.

핵심은 “AI에게 답을 묻는 것”에서 “AI에게 도구를 연결해 일을 맡기는 것”으로 이동하는 것입니다.

AI가 여러분의 문서, 디자인 도구, 사내 시스템과 연결된다면
가장 먼저 어떤 일을 맡기고 싶나요?

직접 설정 문장을 처음부터 쓰기 어렵다면 Codex에게 초안을 만들게 합니다.

    Codex
    [표지] 제목과 부제를 읽습니다. 사람을 복제한다는 뜻이 아니라, 내 기준과 일하는 방식을 AI가 더 잘 참고하게 만드는 과정이라고 먼저 정리합니다.
    [목차] 좋은 프롬프트에서 시작해, Instruction, Context, Memory, Skill, Agent를 이해하고, 마지막에는 Codex에서 실제로 지침과 Memory를 설정하는 흐름이라고 안내합니다.
    [오프닝] AI와 친해진다는 말을 “내가 원하는 일의 기준을 함께 정하는 것”으로 설명합니다. 오늘은 기술 자체보다 기준을 전달하고 저장하는 법에 집중한다고 말합니다.
    [좋은 프롬프트란] 왼쪽과 오른쪽은 같은 요청입니다. 차이는 역할, 목표, 배경, 해야 할 것 / 하지 말아야 할 것, 출력 형식, 확인 기준이 분리되어 있는지입니다. 수강생에게 문장 길이가 아니라 판단 기준의 유무를 보게 합니다.
    [개념 파트 전환] 좋은 프롬프트의 항목들이 실제 AI 도구에서는 지침, 기억, 절차, 역할 같은 형태로 저장될 수 있다고 연결합니다.
    [Instruction] Instruction은 총 주방장이 항상 지켜야 하는 운영 원칙입니다. 손님 요청 확인, 담당 Agent에게 일을 나누는 기준, 하지 말아야 할 것, 마지막 검수 방식을 파일로 남긴다고 설명합니다.
    [Memory] Memory는 총 주방장이 계속 참고하면 좋은 반복 기억입니다. 단골 손님 취향, 자주 쓰는 재료, 선호하는 설명 방식처럼 참고하면 결과가 좋아지는 정보를 남긴다고 설명합니다. 다만 알레르기와 금지 재료처럼 반드시 지켜야 하는 조건은 Memory에만 맡기지 말고 매번 요청에서 다시 확인해야 합니다.
    [Context] Context는 AI와 주고받은 질문, 답변, 수정 요청, 첨부 자료가 이어진 대화의 흐름입니다. AI에게 질문하고, 답을 받고, 다시 고쳐달라고 말하는 그 흐름 전체라고 설명합니다.
    [Context Window] 막대 전체가 AI가 한 번에 참고할 수 있는 범위입니다. 대화가 길어지면 이전 내용은 요약되고, 세부 표현이나 작은 조건 같은 내용 일부는 빠질 수 있습니다. 중요한 기준을 다시 적으면 새 요청 쪽 비중이 커진다고 설명합니다.
    [Skill] Skill은 반복 작업 순서를 저장하는 방식입니다. 특정 요리를 만들 때 재료 확인, 도구 준비, 조리 순서, 마지막 맛 확인이 정해져 있는 것처럼 반복되는 순서를 저장한다고 설명합니다.
    [Agent] Agent는 책임을 가진 역할입니다. 화면의 예시는 총 주방장 Agent로, 손님 요청을 확인하고 재료 손질, 양식, 찜 요리 담당 Agent에게 일을 나눈 뒤 최종 결과를 검토합니다. Skill이 절차라면 Agent는 그 절차 안에서 판단하고 위임하는 역할이라고 설명합니다.
    [사례 파트 전환] 이제 개념을 실제 불편함에 연결합니다. AI가 이상하게 군다고 느끼는 순간은 대개 요청, 기준, 기억, 확인 방식 중 하나가 흐려졌기 때문입니다.
    [말 안 들어요] 왼쪽 대화에서 금지 표현이 다시 나오는 장면을 읽습니다. 오른쪽에서는 현상, 문제와 설정 위치, 고쳐야 할 부분 순서로 설명합니다. 핵심은 계속 피해야 하는 표현은 Instruction에 저장하고, 이번 답변에서 중요한 조건은 Prompt에도 다시 써야 한다는 점입니다.
    [까먹어요] 왼쪽 대화에서 비개발자 대상 조건이 약해지는 장면을 읽습니다. 오른쪽에서는 Context, Context Window, Memory를 연결해 설명합니다. 대화가 길어지면 일부 조건이 요약 과정에서 빠질 수 있으므로 중요한 기준은 새 요청 맨 앞에 다시 적어야 합니다.
    [자신 있게 틀려요] 왼쪽 대화에서 확인되지 않은 정보를 확실한 것처럼 말하는 장면을 읽습니다. 오른쪽에서는 Context, Instruction, Skill을 연결합니다. 근거 자료가 없으면 추측이 섞일 수 있으므로 모르면 모른다고 말하는 지침과 확인 체크리스트가 필요합니다.
    [내 기준대로 일하는 AI] 완벽한 사람 복제가 아니라 내 기준으로 일하는 방식을 만드는 것이라고 정리합니다.
    [가능성] Codex와 Claude Code 같은 제작형 AI는 지금 보고 있는 교육자료 같은 문서와 슬라이드, 스프레드시트와 구글 슬라이드 같은 업무 파일, 개인에게 필요한 작은 맥 프로그램까지 만들 수 있다고 설명합니다. 원하는 결과가 분명하면 AI는 답변 창을 넘어 함께 만드는 작업자가 된다고 정리합니다.
    [반복 개선] 처음부터 내 기준대로 일하게 만들 수는 없습니다. 너무 어렵다, 근거가 부족하다, 예시가 약하다 같은 피드백을 지침, 기억, 절차로 옮길수록 다음 결과가 좋아집니다. 하단 카드 4개를 눌러 Codex에게 초안을 맡기고, 설정 완료 응답과 Skill 사용법까지 확인하는 흐름을 보여줄 수 있습니다.
    [지침 작성 가이드] 이 슬라이드 자체도 Instructions를 적용해 만들었다고 강조합니다. 지침은 긴 명령문이 아니라 AI가 반복해서 참고할 역할과 기준이며, 역할, 해야 할 일, 하지 말아야 할 일, 결과 형식, 참고 기준을 나눠 쓰면 AI가 더 안정적으로 같은 방식으로 답한다고 설명합니다.
    [Plugin과 MCP] Plugin은 이미 만들어진 확장 기능을 붙여 쓰는 느낌이고, MCP는 AI와 외부 도구나 데이터를 연결하는 표준 통로라고 설명합니다. Skill과 Agent는 일하는 기준이고, Plugin과 MCP는 실제 도구와 데이터에 닿는 통로라고 구분해줍니다.
    [외부 도구 연결] 문서 데이터 업데이트, Figma 같은 디자인 도구 확장, 반복 업무 자동화, 작은 내부 도구 제작을 예로 듭니다. 수강생들이 “AI로 어디까지 할 수 있나”를 답변 생성이 아니라 업무 도구 연결 관점으로 이해하게 합니다.
    [Figma 플러그인 제작 사례] 먼저 요구사항을 읽습니다. 피그마 화면의 텍스트를 한꺼번에 모아 다국어 key를 설정하고, 적용된 key를 annotation(화면 메모)으로 확인하는 것이 목표였다고 설명합니다. 그 다음 Instruction과 Memory로 작업 기준을 세팅하고, Brainstorm과 Deep Interview Skill로 요구사항을 개발 기획서로 바꿨다고 말합니다. 이후 “이 기획서를 기준으로 Figma Plugin을 만들어줘”라고 구현을 맡겼고, 우측 영상은 그 결과물이라고 연결합니다.
    [Q&A] 질문을 받습니다. 질문이 없으면 “AI가 여러분의 문서, 디자인 도구, 사내 시스템과 연결된다면 가장 먼저 어떤 일을 맡기고 싶나요?”라고 유도합니다.
