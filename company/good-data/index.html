Q1. 이 폴더, 뭐가 문제일까요?

Q1 개선 · 이렇게 정리하면

같은 폴더, 깔끔하게 (옛 버전·빈 파일·잡음 정리, 스캔 → 문서)
  • 최신본 하나만 + 날짜 → 옛 정책 오인용 ✗
  • 한 주제 = 한 파일, 제목만 봐도 내용 파악
  • 빈 파일·잡음 제거 → AI가 헷갈릴 거리 없음
  • 스캔 이미지 → 글자(텍스트)로
❓

Q2. 이 문서들, 뭐가 문제일까요?

둘 다 「비대면 계약」 문서 — 그런데 망가진 방식이 서로 다릅니다
비대면 계약 안내 ■ 내용 계약은 보통 비대면. 상황에 따라 대면도 가능. 외부 직거래는 되도록 ✗ 연락처는 적절한 시점에 공개. ■ 기타 카톡 영상 공유? → 확인 필요 (TODO) 방문 예외는 케이스 바이 케이스.
비대면 계약 관련 정리 저희가 보통 비대면으로 진행하고 있고 그동안 여러 케이스가 있었는데, 대체로는 비대면이 원칙이지만 고객 상황이나 매물 특성, 담당자 판단에 따라 대면으로 진행한 적도 있었고 예전에는 대면이 더 많던 시기도 있었습니다. 외부에서 따로 거래하시는 부분은 회사 입장에서 권장하지 않는 편이고 가급적 정식 절차를 통해 진행하시는 게 좋습니다만 경우에 따라 달라서… 연락처 공개 시점은 계약 의사가 어느 정도 확인된 이후가 일반적이었던 것 같고 다만 케이스마다 조금씩 달라 일률적으로 말하긴 어렵습니다. 카톡 영상 공유나 방문 관련해서도 그동안 논의가 많았는데 결론적으로는 담당자와 협의하여 처리하는 편이며…

정리: 좋은 지식베이스 데이터의 조건

한마디로 — "사람이 봐도 좋은, 잘 정리된 문서"

메타가 있다 — 대상·날짜·작성자
제목·구조가 명확하다 — 한 주제=한 파일
글자로 돼 있다 — 스캔 이미지 ✗

A. 이렇게 쌓으면 — '좋은' 학습 데이터

Q&A만으론 부족했던 이유 → 상황·근거·메타를 붙이면 (보증금 차감 문의)
❌ 좋은 데이터가 아닌 이유
맥락 없음 → 같은 질문이어도 맥락에 따라 답변이 달라지는데 AI는 해당 정보를 알 수 없음
근거 없음 → AI는 정확한 이해 없이 암기함 → 다른 케이스서 지어냄(환각)
옛 정책 포함 → 옛 정책으로 답변한 데이터도 포함되어 있음 -> 틀린 걸 학습
✅ 이렇게 쌓으면 (보증금 문의 재구성)
유형보증금 > 공제 분쟁 · 1:1 문의
상황진행중인 계약 있음(이용중) · 임차인(홍길동) · 카드 결제 · 계약 이력 3건 · 분쟁이력 없음
문의"보증금은 언제 돌려받나요?"
답변계약이 종료되고 보증금 반환 신청 버튼을 누르면 됩니다.
근거사전 동의 없는 차감 불가 — 「보증금 공제 절차」 기준
메타2026-06-07 · 정책 v2026.6 (현행 ✔) · 개인정보 마스킹

정리: 좋은 모델 학습 데이터의 조건

한마디로 — "답만이 아니라, 상황·근거까지 잘 쌓은 사례"
1 가르칠 만한가
정확하다 — 틀린 답은 뺀다 (AI는 준 대로 다 배움)
최신이다 — 바뀐 과거 답은 고치거나 제외
2 맥락이 있는가
상황이 있다 — 계약·사용자 맥락 함께 저장
근거가 있다 — 출처 규정을 연결
3 고르고 깔끔한가
편식 없다 — 골고루 (🔁 아마존 채용 AI)
깔끔한 쌍이다 — 질문·답 구분 명확 · 톤 일관

주의 ② AI는 늘 의심하자

에어캐나다 챗봇 사건 (2024, 캐나다 법원 실제 판결)
고객이 챗봇에게 "유족 할인 환불 되나요?" → 챗봇이 실제 정책엔 없는 환불을 "된다"고 답변. 그 말 믿고 구매 → 환불 거부 → 소송 → 법원: "챗봇이 한 말도 회사 책임" → 항공사 패소·배상.
💡 챗봇이 틀린 이유 = 옛·모순된 정책 데이터.
